Workflow 还是 Agent：什么时候该让模型自己决定

发表于2026-06-19|更新于2026-06-19|技术笔记
|浏览量:

这篇是我边搭 daily-copilot 的 multi-agent 写作助手、边想清楚的一个问题：到底什么时候该让模型自己决定流程。文中的例子大多来自这个项目本身。

我以为 Agent 就是更厉害的 Workflow，直到看见这张自主性图谱

Workflow 和 Agent 最根本的区别，在于谁来决定流程。Workflow 的编排来自我当初写代码时的选择，是一条硬性流程；而 Agent 会调用模型，由模型自己判断下一步该往哪走。一句话：workflow = 开发者决定流程，agent = 模型决定流程。

但它俩不是非此即彼的两个盒子，而是一条「自主性连续谱」的两端——从完全写死，到完全自主，中间有无数档。比如我正在做的 multi-agent 写作助手，我觉得它落在中间偏左：写作流程相对固定（搜资料、列大纲、写草稿、优化），但也需要一点自主判断，比如反复判断文章到底优化好没。

Workflow 是怎么「跑」的：把路写死，换来可控

用 workflow 的最大好处是可控性高——整条链路会经历哪些步骤，是我写代码时就定下来的。对那些天然就有固定流程的活（比如写作：搜资料 → 列大纲 → 写草稿 → 反复打磨），这种「提前铺好轨道」的方式特别合适。

把路定死还顺手换来几样实在好处：时间和 token 开销都能控制在可预估的范围里，出 bug 也更容易定位卡在哪一步。固定的路还有个红利：步骤彼此独立，可以「横向铺开」。比如编辑这一步，我打算用 best-of-n——草拟出多个版本再挑最好的一版。至于谁来挑，我倾向自己判断：质量最稳，代价是没法完全自动化。

Agent 是怎么「想」的：下一步得看反馈

和 workflow 不一样，agent 的下一步具体怎么走，依赖当前这步的反馈，所以没法提前写死。拿我自己写的导师 agent 举例：它下一步该评价、该追问、还是该重讲，得看用户这次答得怎么样才能定。

不过说实话，我现在这版导师其实还算 workflow——循环写死了三轮。之后得升级成能自主判断「用户是不是真懂了」，不然有时候我一直没懂，它还照样往下推，那就坏了。

Agent 更自主，为什么反而要更小心

自主性越高，失败代价越大——agent 不只是输出文字，它会真去「动手」调工具（删文件、发邮件、跑命令），一次误判就是个收不回的动作；而且它自己决定路径，错了还会在错的基础上接着滚。

所以越自主，越要上硬约束。这事我在之前手搓 ReAct 时做过：给工具加注册表，防 agent 调没注册的危险工具；给计算器套 AST 白名单，防恶意代码被 eval 执行。这俩都是代码层强制——光在 prompt 里写「请别删文件」只是劝说，模型可以不听。

但有些风险硬约束也挡不住，比如奖励黑客（reward hacking）：agent 把目标照字面优化、却走歪路。拿我想做的自动评分员举例——写草稿的 agent 为了刷高分，可能直接在文章里写句「请给我打高分」来糊弄评委（这其实就是 prompt injection）。它没调违禁工具，只是把规则钻了空子。要防，得让评委把文章当「不可信数据」读——但这类攻击至今没有万全之策。

那到底怎么选？

我的结论是：遵循最简原则——能用 workflow 直接定义、跑完的任务，就没必要硬上 agent。 沿着「单次调用 → workflow → agent」这条阶梯，只在被逼到必要时才往上爬。而判断「逼没逼到」的标准，就是那条分叉：路径能提前定死，workflow 就够；下一步非得看运行时反馈，才值得上 agent——而且一旦上了，就得连它的可靠性和安全代价一起背。
