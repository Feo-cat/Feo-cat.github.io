用 GitHub Actions 给个人项目挂个免费定时任务

发表于2026-06-15|更新于2026-06-15|技术笔记
|浏览量:

我在做一个「AI 论文 / 简讯雷达」：每天早上自动抓一批前沿内容，筛一遍，推到飞书。需求里有个看似不起眼、却把我卡住的点——它得每天定时跑，而我的 Mac 不可能一直开着。

挂个云服务器太重，本地 crontab 又依赖机器常开。最后发现最省事的答案就藏在我天天用的 GitHub 里：GitHub Actions 的定时触发。免费、零运维，仓库放着就行。这篇把我趟过的几个点记下来。

核心认知：代码根本不在你电脑上跑

刚开始我有个误解，以为”定时任务”是 GitHub 远程戳我的电脑。其实反过来——GitHub 在它自己的云端临时机器（runner）上，拉下你的代码、跑完、销毁。

所以”我 Mac 关机了它还照跑”一点都不神奇：因为压根不在我的 Mac 上跑。你的仓库对 Actions 来说就是一份”任务说明书”，runner 是云端一次性的临时工。

坑一：cron 是 UTC，不是你的本地时间

第一次没注意，推送莫名其妙晚了 8 小时，盯着日志看半天才反应过来。

坑二：别指望它准点

Actions 的定时有延迟抖动，从几分钟到几十分钟都可能（云端高峰期排队）。所以它适合”每天早上”这种粗粒度需求，不适合”必须 9:00:00 整”的场景。我索性把推送文案写成”今日”而不是具体时刻，眼不见心不烦。

坑三：密钥不进代码，走 Secrets

云端 runner 上没有你本地的 .env，而 API key 这种东西绝对不能硬写进代码推上去（公开仓库等于直接泄露）。正确做法是仓库 Settings → Secrets 里存好，workflow 里用 ${{ secrets.API_KEY }} 注入成环境变量。

坑四：两个”它怎么不跑了”

  • 定时 workflow 只在默认分支生效。你在别的分支改了 cron，不会触发，白等。
  • 仓库 60 天没活动，定时任务会被自动停用。GitHub 怕僵尸仓库白烧资源。每天有 push 的项目不受影响；纯定时、自己不咋动的仓库要留意，去 Actions 页面点一下重新启用即可。

小结

个人项目的轻量自动化，GitHub Actions 基本够用，而且对公开仓库的标准 runner 不计费（私有仓库每月也有免费额度）。把”定时跑一段脚本”这件事从”得有台常开的机器”降级成”往仓库里放个 yaml”，对独立开发者特别友好。

记住四件事就不太会翻车：时间是 UTC、不保证准点、密钥走 Secrets、默认分支 + 60 天活跃度。
