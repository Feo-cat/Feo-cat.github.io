用 GitHub Actions 给个人项目挂定时任务

发表于2026-06-15|更新于2026-06-15|技术笔记
|浏览量:

我在做一个每天清晨自动运行的 AI 雷达：抓取前沿论文与简讯，筛选之后推送到飞书。它的逻辑并不复杂，真正卡住我的是一个很基础的问题——它必须每天准时运行，而我的电脑不会一直开着。

通常的答案是租一台服务器，或者依赖本地的 crontab。但对一个个人项目而言，前者太重，后者又把”可靠”绑死在一台需要常开的机器上。后来我发现，我每天都在用的 GitHub，本身就给了答案。

它不在你的电脑上运行

理解 GitHub Actions，要先放下一个直觉上的误会：定时任务并不是 GitHub 远程唤醒你的电脑去执行什么。恰恰相反——GitHub 在它自己的云端，用一台临时的虚拟机，拉取你的代码、运行、然后把这台机器销毁。

一旦接受这件事，”电脑关机了它也照常运行”就不再神奇，而是理所当然：因为它从一开始就没打算在你的电脑上运行。你的仓库对它而言只是一份说明书，真正干活的是云端那台用完即弃的机器。

这个视角的转变，比任何一行配置都重要。

一份说明书的样子

你要做的，是在仓库里放一个 YAML 文件，告诉那台机器：什么时候醒来、准备什么环境、运行什么命令。

推上去，它就开始按时运行了。剩下的，是几件需要想清楚、而非死记的事。

时间属于世界，不属于你

cron 里的时间是 UTC，不是你所在的时区。我想要北京时间清晨九点，于是写下 0 1 * * *——UTC 一点，正好对应北京九点。这不是一个陷阱，而是一个合理的默认：当机器分布在世界各地，它们只能共享一个时间基准。

与之相关的是，定时触发并不精确。云端在高峰期会排队，实际运行可能比你设定的时刻晚上几分钟到几十分钟。所以它适合”每天清晨”这样的节奏，而不适合”九点整”这样的承诺。我索性把推送文案写成”今日”，让这点模糊变成一种从容。

密钥属于保险箱，不属于代码

云端那台机器上没有你本地的 .env，而 API key 这类东西也绝不该写进代码——公开仓库里的一行密钥，等同于公开它。GitHub 为此准备了 Secrets：你把密钥存进仓库设置里那个加密的保险箱，在 YAML 里用 ${{ secrets.API_KEY }} 取用。代码这边则统一从环境变量读取，本地由 .env 兜底，云端由 Secrets 注入，两处共用一套逻辑。

还有两处安静的边界值得知道：定时任务只在默认分支上生效；以及，当一个仓库连续六十天没有任何活动，它的定时任务会被自动停用——这是 GitHub 对僵尸仓库的一种克制。每天都有提交的项目不会遇到后者，但若你的仓库只靠定时器独自运转，记得偶尔去唤醒它。

写在最后

GitHub Actions 把”定时运行一段代码”这件事，从”你需要拥有一台常开的机器”，变成了”你只需要往仓库里放一个文件”。对个人开发者来说，这种简化是温和的：它不要求你懂运维，也不向你收费（公开仓库的标准运行环境不计费），只是安静地，每天替你把那件小事做完。

需要记住的并不多——时间以 UTC 计、不保证准点、密钥走 Secrets、定时只认默认分支。剩下的，交给那台你看不见的机器就好。
