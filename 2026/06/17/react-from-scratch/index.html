不用框架，从零手搓一个 ReAct Agent

发表于2026-06-17|更新于2026-06-17|技术笔记
|浏览量:

代码在我的 daily-copilot 项目里：experiments/react_warmup.py（约 180 行，无任何 agent 框架、不用 SDK 原生 tool-use）。

0. 引子：ReAct 是什么

我给自己出了道小题：让一个大模型回答「距离项目收尾（2026-08-21）还剩几周零几天？」。

听着简单——直接问它不就完了？可你真去问，会撞上两个坑：它根本不知道”今天”几号（模型有知识截止日期，读不到你的系统时钟），而且就算把日期喂给它，它的心算也靠不住（大模型本质是在”猜下一个字”，不是在做精确运算）。

也就是说，光让模型”想”不够，得让它能”动手”——查一下今天几号、调个计算器算一下。而怎么让一个只会生成文字的模型”动起手来”，正是 ReAct 要解决的问题。

ReAct = Reasoning + Acting（推理 + 行动）。它让模型在”想一步（Thought）”和”做一件事（Action）”之间反复横跳，每做一件事就拿到一条真实结果（Observation），再据此想下一步：

这篇文章里，我不借任何 agent 框架、也不用 SDK 自带的”函数调用”，只用最朴素的三样东西——一段提示词约定 + 一点字符串解析 + 一个 while 循环——把这套”心跳”亲手搓出来，看看一个 agent 到底是怎么转起来的。全部代码不到 180 行。

1. 工具与安全：LLM 的输出是「不可信输入」

eval() 做的事其实很简单：把一个字符串当成 Python 语句直接执行。算 2+3 它没问题，但它分不清那是一道算术题、还是别的什么东西——只要传进去的是合法的 Python，它就照单全收地跑。而在 ReAct 循环里，这个字符串是大模型生成的。模型大多数时候老实给我算式，可只要哪一次它的输出因为某些原因变成了一段高危操作，eval 就会毫不犹豫地替它执行——那就出大问题了。这种「把外部输入当代码执行」的口子有个正式名字：RCE（远程代码执行），而且它未必是意外，也可能是有人故意构造一段输入来打你。

那 _safe_eval 怎么解决？思路是：不把字符串交给 Python 去「执行」，而是先用 ast 把它解析成一棵语法树。拿到这棵树之后，我只放行两类节点——数字和我认得的运算符；一旦遇到别的东西，就直接拒绝。能跑通的就只剩下纯算术，夹带的代码在解析这一步就被挡在门外。说白了这是一种白名单思路：只许做安全的、其余全拒；它的反面是列一串危险操作去禁（黑名单），可黑名单永远堵不完，白名单才天生安全。

同样的思路，往上放大一层，就成了 agent 的能力边界。我们规定：agent 只能调用注册表里登记过的工具，没登记的一概不认。于是哪怕模型在 Thought 里突然冒出「我要删个文件」的念头，只要 TOOLS 里压根没有「删文件」这一项，这念头就停在嘴上——它想得到，却做不到。

那为什么不能只在 prompt 里叮嘱一句「别干坏事」就了事？因为这是大模型底层原理决定的：无论我们在 prompt 层面怎么恳求，模型总有绕过边界的风险。所以安全不能押在「模型愿不愿意听话」上，而必须落到代码里：_safe_eval、TOOLS 注册表，都是在执行那一刻用函数硬生生卡住模型的输出。一句话：prompt 是劝说，代码才是强制。

2. 提示词即协议：靠「说服」而非「强制」

在写循环之前，得先和模型「约法三章」——这就是 system prompt 干的事。它本质是一份自然语言说明书：用大白话告诉模型有哪些工具能用、该按什么格式回话（还可以附一个参考示例，让它照葫芦画瓢），以及几条行为规矩——比如一次只走一步、**别自己编 Observation**（这条和第 4 节的 stop_sequences 前后呼应）。

那模型凭什么听这段文本的话？因为它在训练阶段被做过指令微调（instruction-tuning），被专门训练成「偏好遵守指令」，所以大多数时候它会照着格式来。但关键在于：这只是大幅降低了它不听话的概率，并不能完全约束它。换句话说，prompt 给的是一种「软约束」——它是说服，不是强制。这也正是为什么真正要命的地方（别让它跑任意代码、别让它编造观察结果）不能只靠 prompt 叮嘱，而要像第 1、4 节那样落到代码里、用硬约束焊死。

还有个工程上的小讲究：prompt 里那份工具清单，不是我手写死的，而是直接从 TOOLS 注册表自动拼出来的。好处是只有一个事实来源——加一个工具，我只改注册表这一处，prompt 里的清单自动跟着更新，绝不会出现「注册表里有、prompt 里却漏了」这种对不上的 bug。这就是 DRY（Don’t Repeat Yourself） 原则。

3. 解析的脆弱：用正则从文本里抠结构

那怎么从模型这段自由文本里抠出工具名和参数？最朴素的办法就是正则：按 Action: / Action Input: 这样的标记去匹配后面跟的内容。可问题在于，模型的输出是不可控的——它大概率会按格式来（上一节说的指令微调），但只要哪次稍微跑偏：把冒号写成全角 ：、换成破折号、或者多空了一行，正则就可能匹配不上、抠出个空。用正则去解析自然语言，天生就是脆的。

那解析失败了怎么办？不能让程序直接崩。 parse_action 抠不到时会返回一个 None——这个 None 与其说是「没结果」，不如说是一个「解析失败」的信号。主循环收到它，不崩，而是往历史里塞一句「你格式不对，请按规定格式重写」，然后 continue 进下一圈，让模型重新输出一轮。这恰好兜住了上一节埋的伏笔：prompt 是软约束、会失败，所以代码这边得备好「失败了就让它重来」的退路。

当然，这套「正则抠 + 抠不到就重来」的做法，终归有点笨重。生产环境里的真正解法，是直接用原生工具调用（tool-use）：在定义工具的时候，就用 schema 硬性规定好模型的输出格式——相当于先把输出的骨架（工具名、每个参数的字段）划定死，模型只需要往里填空。这是一种硬约束，根本不存在「格式跑偏、正则抠不到」这一说，自然也就不需要解析和重试的兜底了。

4. agent 的心跳：run_react 主循环 ⭐

run_react 的核心是一个循环，我们来走一圈。开头，把用户的问题发给模型；模型推理一番，得出这一步该调哪个工具、该传什么参数（也就是它吐出的 Thought 和 Action）。我们解析出工具名和参数，真正去执行那个工具，拿到结果。最后一步是关键：把这个结果拼回历史消息里，下一圈连同整段历史一起再发给模型。于是模型看着「上一步我查了什么、得到了什么」，接着想下一步——如此一圈圈转下去。

这里藏着一个容易被忽略、却特别重要的事实：大模型本身是没有记忆的。每一次调用，都是把到目前为止的整段历史一股脑发过去、再等它输出；它表现得「记得」上文，纯粹是因为我们每轮都把历史重新喂了一遍。一旦哪次偷懒、只发这轮的新消息，模型就会因为丢了大量上下文而答出莫名其妙的东西。记忆不在模型里，在我们那个 messages 列表里。 不过这也有代价：既然每圈都重发整段历史，轮数越多、发得越多，就越来越贵，迟早还会撑爆上下文窗口——这正是后面要做「记忆裁剪」的起点。

循环里还有一行不起眼、却是全场最硬的代码：stop_sequences=["Observation:"]。它让模型生成到 Observation: 这个词之前就强制刹车。为什么非拦不可？因为模型是个续写机器——你给它 Thought / Action / Observation 的格式，它写完 Action，很可能手一滑，把 Observation 也一并替你编出来。可这个「观察结果」是它脑补的，工具根本没真跑；一旦让这种假结果混进历史，后面全错——这等于把 ReAct「让模型接触真实世界」的意义整个废掉了。所以我们在 Observation: 前一刻把它叫停，把这个位置空出来，留给代码填真实的工具输出。

那循环怎么结束？它有两个出口。一个是正常出口：模型觉得自己想明白了，直接吐出 Final Answer:，不再请求调用工具——循环就此收工。（有意思的是，Claude Code 这类生产级 agent 也是这个逻辑：模型不再发起工具调用，这一轮就算结束。）另一个出口是道保险：MAX_STEPS，一个最大步数上限。它防的是模型卡死——万一它一直 Thought → Action → Observation 兜圈子、就是绕不出 Final Answer，没有这个上限，循环就会无限转下去：白白烧掉大量 token，还把一堆没用的中间步骤塞进上下文、污染模型的注意力。所以它本质是个熔断器：转够 N 圈还没结果，就强制断电。

5. 跑起来 + 收尾：prompt 说服，代码兜底

整条轨迹把前面讲的东西全演了一遍：模型不知道今天几号（知识截止），于是第一步先 today 把盲区补上；它不敢心算，把 65 // 7、65 % 7 都丢给计算器；每一步只走一格、拿一个真实 Observation 喂回去；直到它觉得够了，自己吐 Final Answer 收工。180 行代码，没有任何框架，一个 agent 的「心跳」就这么转起来了。

写到这儿回头看，我对「agent」这个词的感觉彻底变了。以前它对我又熟又玄——天天听、也在用，却从没想过底层是什么。真把它拆开、一行行读完才发现：所谓 agent，底子上不过就是一个循环 + 工具调用——模型想一步、调个工具、看一眼结果，再想下一步，如此而已。没有魔法。可恰恰是这么朴素的一个骨架，撑起了那么能干的东西，大道至简大概就是这个意思。

而这趟从零手搓，给我的不只是「看懂了」。这套「想—做—看」的循环，还有一路上那条软约束靠说服、硬约束靠代码的取舍，是可以直接搬走的——接下来我要做的 multi-agent 写作助手，骨子里还是这一套。读懂了最小的那个，再大的 agent 也就不慌了。
