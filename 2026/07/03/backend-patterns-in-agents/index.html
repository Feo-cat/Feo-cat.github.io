实习里学的状态机和工厂，在我的 agent 代码里复活了

发表于2026-07-03|更新于2026-07-03|技术笔记
|浏览量:

这是一篇工作日晚的短文。白天在公司实习写 Go 后端，晚上回来搓 daily-copilot 的 multi-agent 写作助手，两边代码看多了，我在 agent 这边认出了两个实习里的老朋友。涉及实习的部分已脱敏，只留下模式本身。

我以为这些模式会留在公司代码库里

做 agent 项目之前，我隐约觉得业务后端那套——状态流转、注册分发——和「让模型自己决定下一步」的世界隔着次元壁。真写起来才发现，agent 代码里让我最踏实的两块，恰好是实习里最眼熟的两个模式。而且它们在这边不是锦上添花，是不用就出事。

状态机：换了个对手，还是那台机器

实习里我打交道最多的状态机，是订单/工单类的状态流转：状态有限，哪个状态允许转到哪个，写死在代码里；条件不满足的转移直接拒绝——前置环节没走完的单子，谁来也别想把它标成「完成」。当时我把这当业务代码的常识：状态机守的不是流程图好看，是数据不被写坏。

我的写作助手里有个「导师」agent：逐节先讲再问，拿到我的回答，由模型判断我懂没懂，决定继续教还是收尾。拆开看，这就是一台小状态机：「讲→问→答」是自循环，出口只有两个——模型输出【本节已掌握】暗号，或者轮数触顶熔断。

关键在那行放行代码：if DONE_TAG in out and records。转移由模型建议，由代码裁决——暗号只是模型的提议，后面的 and records 才是转移守卫：一轮都没问过就宣布掌握，属于非法转移，拒绝。这跟订单系统拒绝「未支付 → 已完成」是同一件事。

不同的是对手换了。后端状态机防的是脏数据和乱调用；agent 状态机防的是模型的不确定性——它可能第一轮就急着收尾，也可能永远不收。所以除了守卫，还得配 MAX_ROUNDS 这个兜底终态，保证这台机器无论如何会停。后来看到 LangGraph 干脆把 agent 建模成 StateGraph——殊途同归。

工厂与注册表：同一张表，两个世界

另一个高频老朋友是 handler 注册表：各类处理器按 key 注册进一张表，运行时查表分发；新增一种类型，只需注册一个新 handler，分发逻辑一行不改。Go 里 map 加 init() 注册的写法，见多了就成肌肉记忆。

agent 这边，同一张表换了个名字叫工具注册表：模型在输出里报工具名，我的代码查表执行，没登记的一概不认。写《手搓 ReAct》时我把它当安全白名单讲——模型「想得到，做不到」；换回后端视角，它就是那张最普通的分发表：注册即接入，查不到即拒绝。一张表，同时买到扩展性和能力边界。

再往上一层，agent 本身的组装也是工厂味儿的：每个 agent = 一份角色 prompt + 一档按需配置的模型（机械步用便宜的，关键步顶好的），全按配置表创建。之后研究员的「服务端搜索 / 自研检索」双实现降级也靠它：同一个接口，创建时按可用性挑实现。

诚实刹车：复活的前提是真有问题

这不是「设计模式放之四海而皆准」的爽文。这两个模式在 agent 里复活，是因为先有真问题：循环可能不停、工具可能乱调、判断可能失灵。新变量也扎眼：后端状态机的转移条件是确定的（字段、事件），agent 里最关键的那个转移判断（「他懂没懂」）交给了一个概率模型。所以守卫和熔断在这边不是防御式编程的好习惯，而是必需品。这也是我之前那句「prompt 是劝说，代码才是强制」的另一面。

所以后端功底没有作废，它只是换了个对手：从不可信的外部请求，换成了不可信的模型输出。模式没变，变的是你信不过谁。
