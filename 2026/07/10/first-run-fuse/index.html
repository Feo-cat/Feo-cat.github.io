研究员第一次真跑，就把熔断撞响了

发表于2026-07-10|更新于2026-07-10|技术笔记
|浏览量:

工作日晚的短文。这周我把写作助手的研究员从「服务端 web_search」换成了手写的 client-side 检索循环（起因是一把失效的 API key，完整改造记之后另写）。本文只记它第一次真跑的 168.8 秒，和一条被撞响的熔断。代码在 daily-copilot。

一组数字

选题是很小的一个：「什么是 prompt caching」。研究员跑完，账单如下：耗时 168.8 秒，LLM 调用 5 次，搜索 12 次，tokens 18,578 进 / 1,873 出。

这里面最有意思的是「5 次调用」。我的检索循环给了轮数上限（4 轮）：每一轮，模型可以要求搜索，我的代码去调搜索引擎、把结果塞回去；模型觉得够了，就该交出素材简报。结果 4 轮它全在喊「还要搜」——平均每轮甩给我 3 个查询词——第 5 次调用是我收走工具、强制收尾的那一下。

也就是说：第一次真跑，它就贪到了熔断。

顺带一提成本：这一跑走的是便宜档模型加免费搜索，token 账单折算下来不过几分钱——真正贵的是时间。168.8 秒里大头不是模型思考，而是 12 次搜索在我的循环里串行执行；每次两三秒，攒起来就是一分钟量级。这暴露的是我循环实现的朴素，不全是模型的锅。

熔断为什么不是装饰

上一篇写状态机时，我说自主循环要配一个「兜底终态」，当时是设计上的谨慎。这次它变成了真实数据：没有那个上限，我不知道它还想搜几轮——自主性意味着模型决定下一步，也意味着你无法预知它要走几步。

所以出口必须是代码强制的，而且我发现光「不再理会搜索请求」还不够——熔断后要把工具从请求里拿掉，模型才会死心，用手头已有的信息就地成稿。工具还在场，它就总想再来一次。这和劝说无关，和结构有关：你给它什么选项，它就在什么选项里打转。

「贪」是 bug 吗

我觉得不是。研究员的职责就是查证，多搜不算错——错的是无界。这正是从固定流程走向自主循环的代价具象化：workflow 的成本在写代码时就定死了，agent 的成本是一个分布，长尾可以很长。

工程上的答案不是禁止自主，而是给分布加上界：轮数封顶、每次搜索的条数封顶、必要时再加超时；再往后还有并行化搜索、给「要不要再搜一轮」加代价提示这类真正的优化。这些我暂时一个都没动——故意留着这组「贪到熔断」的原始数字，它是之后所有调参和优化的 baseline，也是「自主不免费」最诚实的一张收据。

收尾

之前那句「转移由模型建议，由代码裁决」，这周从格言变成了账单：168.8 秒，12 次搜索，一次熔断。

先给每个自主循环装一个它挣不脱的出口，然后再谈自主。
