我把研究员从服务端搜索，手搓成了 client-side ReAct 检索循环

发表于2026-07-19|更新于2026-07-19|技术笔记
|浏览量:

这是写作助手系列的核心篇：为什么、怎么做、代价是什么，一次讲完。有个套娃的事实先交代：这篇的初稿素材，是我用自己的写作助手的「making-of 模式」对我自己做的一场访谈问出来的——而这个模式本身，就诞生于写这篇时的一次翻车（见第四节）。代码在 daily-copilot。

研究员变手搓：为什么我把 server-side 搜索干掉了

先说清研究员是谁：写作助手里负责联网查资料的那个 agent。它原来的实现很省心——代码里只声明一行 Anthropic 的服务端 web_search 工具，搜索和循环全在人家服务器里代跑，我这边一行循环都没有。

当时想着 Anthropic 的 API key 充值流程相对麻烦，本来就有换到其他模型的打算。而且我很清楚：服务端 web_search 是 Anthropic 专属的，必须用它家的 API 才能调，换模型就一定会丢掉这个工具。所以做切换之前，就同步决定把它换成手动实现的版本——顺手把整套 LLM 调用迁成了 OpenAI 兼容、配置驱动，base_url/key/model 全从环境变量读，之后换 provider 不用改代码。

其实这件事在项目刚开始的时候我就想过了：最早就想自己手搓一个 client-side 的可降级检索作为熔断兜底。当初没一上来就做，主要是想先把基础链路跑通，让整个流程能端到端地走起来。所以 401 那把火与其说是逼我改，不如说是把 backlog 里排着队的事提前推上了台。

三件套 + 一个 while：client-side ReAct 怎么做出来的

拆开看就三件套，加一个自己写的 while。

一是用 OpenAI function calling 声明一个 web_search 工具，参数就一个 query——模型只负责「说要搜什么」。二是搜索后端用 ddgs（DuckDuckGo），免费、免 key。三是循环本体：模型回 tool_calls，我的代码解析出 query、去调 ddgs、把结果作为 role=tool 塞回 messages，再请求一次——直到模型不再要搜、交出素材简报。手感不是凭空来的：之前练手写过纯文本版的手搓 ReAct 和原生 tool-use 两个实验脚本（《不用框架，从零手搓一个 ReAct Agent》），这回是把那套肌肉记忆用在真工具上。

ddgs 的选型过程，老实说：没有选型。当时是和 AI 结对，它先写了 ddgs 版，我 review 时认下的——决定性因素就一条，免费免 key。刚被 key 失效坑到停摆的人，对「再添一个凭据依赖」有生理性排斥。这个零成本选择后来暴露的账也都记着：串行搜索慢；偶发限流，所以搜索函数把异常整个兜住、失败时返回一句「搜索失败，换个查询词」的文本让模型自己重试；还有个小彩蛋——这个包 2024 年从 duckduckgo_search 改名 ddgs，我代码里至今留着两个名字都兜住的 import，算白嫖生态的隐性成本。

循环配了双保险：轮数熔断（默认 4 轮）+ 熔断后强制收尾。这里有个设计点值得说清：熔断后光「不再理会搜索请求」是不够的，要把工具定义从请求里整个拿掉，模型才会死心、用手头信息就地成稿。这不是实验发现，是写循环时的结构性推理——给它什么选项，它就在什么选项里打转；想让它停，就把选项收走。首跑那次它 4 轮全在要搜、第 5 次调用收走工具加一句明示才交稿，算是验证了「收走」有效；但「只忽略、工具还在」的对照版我从没跑过，这句照实说。有意思的是「错误消息」这个手法我在另一个场景真用了——搜索失败时回错误文本，那是想让它继续用工具时的做法。对照起来一句话：想让它重试，给错误文本；想让它死心，收走工具。两种机制，服务两种意图。

首跑账单 168s：为什么贵的是时间，不是 token

我最直接的观察来自终端那十几行字。运行时屏幕上一行接一行蹦出「🔎 研究员搜索：…」，我眼睁睁看着它一条条往外跳，总共 12 行，等了将近三分钟才停下来。当时的感受很直观：模型完全没打算收手，一轮接一轮地搜，直到第 4 轮结束还没有停止的迹象。实际的轮次结构我是事后倒推才理清楚的——计数器显示一共 5 次 LLM 调用，而我设的上限是 4 轮，说明第 5 次调用就是熔断触发后强制收尾的那一刀。再把 12 次搜索摊进 4 轮，一除就算出来了：平均每轮 3 个搜索 query。

说实话，我根本没预料到第一跑就会撞上熔断。熔断对我来说是防御性设计，骨子里的思路来自《实习里学的状态机和工厂》里反复强调的「兜底终态」——你得给系统一个必然会停下来的保险。我把它放进去，更多是出于设计上的谨慎，觉得「万一失控了总得有人拉闸」。没想到这种谨慎在第一次真实 run 里就变成了真实数据，这也正是后来《研究员第一次真跑，就把熔断撞响了》那篇标题的来源：熔断不是装饰，第一次跑就直接用上了。

至于 12 次串行搜索到底各花了多少秒、为什么它们会成为绝对大头，我得老老实实承认：这次测量里分项计时没做，这是个已知的窟窿。我现在只能告诉你，那将近三分钟的等待里，token 费用折算下来也就几分钱——真正让人心疼的是时间。「大头是串行搜索」是我当时在现场冒出来的判断，在把搜索耗时单独拆出来之前，它只能算推断，不能算严格结论。分项计时已经排进优化 backlog。

文本歧义翻车：研究员把我说的 client-side 搜成了浏览器端

7 月 14 日我照常跑了写作助手，研究员上网搜完，策划输出了一份提纲。我把它贴进和 AI 结对的会话里，review 到一半就直接喊停了——那份提纲围着的全是 WebLLM、WebGPU、transformers.js，这些词我一个都没用过，完全答不上来。

当时我给的只有一句选题串，研究员手上没有任何项目上下文。它上网去搜「client-side」，公网上这个词的主流含义就是浏览器端，于是它按这个含义自信地查了一堆前端 Web 推理的东西回来。这是一次典型的文本歧义翻车，但对我而言它只是个警报：它把纯网搜的脆弱性清清楚楚摆在了面前——研究员没有项目上下文，就必然按通用含义理解术语，哪怕这个词在我代码里的意思完全不同。

不过，让我最终断定必须跳过研究员、直接喂手写素材的，不是歧义，而是另一层东西：类别错误。我要写的 making-of 素材，全在我自己的脑子和本地代码里，网上根本没有。就算研究员完美消歧，它搜回来的也只能是相关技术的通用介绍，而不是我实际踩坑、调试和做设计选择的过程。歧义还能用改措辞、喂 README 这类补丁去缓解，类别错误是补丁救不了的——这才是我给写作助手加「making-of 模式」的根据：跳过研究员、直接喂手写素材，再把策划和导师的提示词从「教学生」翻成「访谈亲历者」。顺带，那份手写素材的第一节至今就叫「名词校准（防歧义，这次翻车的根源）」——疤痕直接长在了素材的开头。

只有「下一步依赖上一步反馈」才值得上循环

这个分界线还真不是我坐在那儿一步步推演出来的，更像是系统自己「长」成这样的。判断标准就一条，我在《Workflow 还是 Agent》里也写过：下一步是不是依赖上一步的反馈。研究员明显满足——它下一个搜索词完全取决于上一轮搜回来的内容，步数提前没法确定，所以必须上循环，还得配熔断。策划、编辑、终审就不一样了：它们的输入都是一次性给齐的——策划拿选题加素材，编辑拿完整的问答记录，终审拿整篇草稿——中间不会突然冒出新信息。遇到失败也就是「产出不好」，整个调用重跑一次就是全部的纠错手段，这时候上循环买不到实际好处。终审的代码里我还特意留了注释：哪天做成「作者回应→再审一轮」，反馈依赖一出现，再把它升级成循环。

还有一个容易被漏掉的角色——导师其实也是循环。下一轮追问完全取决于我上一答，只不过反馈源从工具结果换成了人，所以它被做成了「判定掌握 + 轮数熔断」的双出口状态机。

你要问有没有试过把某些角色强行循环化然后发现不对劲？代码上没这么折腾过，但写作上有一次非常真实的打脸：这篇博客最早定的选题就叫「我把每个 agent 都手搓成了 ReAct 循环」，后来被现实完全证伪，只好把选题废掉——真实跑出来的系统里，只有研究员被 401 逼成了循环，其余都保持单次调用。我本来想交一篇全 ReAct 化的战报，最后写成的却是「为什么我没有全 ReAct 化」。是系统本身的反馈告诉我哪里需要循环，不是我挨个试出来的。

至于 server-tool 和 client-tool 的取舍，我不觉得跟「要不要循环」是同一套判断，它们是两刀。第一刀决定这里要不要循环，看有没有反馈依赖；第二刀决定已经存在的循环放在谁家，看的是控制权——研究员的循环在 Anthropic 时代也存在，只不过跑在人家机房里。要不要搬回家，得看它是不是你需要掌控的核心资产、需不需要扛过换 provider 的风险。两刀背后是同一条原则：复杂度只花在有回报的地方。循环的价格我首跑就付过了：workflow 的成本在写代码阶段就基本定死，循环的成本是个分布、长尾很长，还得配熔断——在根本没有反馈依赖的地方硬上循环，等于白买一份不可预知。

收尾

一把失效的 key，逼我把一个「想做很久」的东西从 backlog 里拽了出来；一次歧义翻车，又逼着这套工具长出了第二种形态。回头看，这条 client-side 循环买到的三样东西——provider 无关、循环在自己手里、熔断规则自己定——每一样都是在某次翻车里被验证值钱的。

访谈初稿的过程里，我的访谈员还两次揪出了我素材里写过头的句子（一处把设计推理写成了「实验发现」，一处把估算写成了实测）——用自己的工具审自己的叙事，这个故事值得单独写一篇，先记账。
